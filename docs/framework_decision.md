# Framework Decision

## Decision

현재 thin_wallet_club 프로젝트는 CrewAI, AutoGen, LangGraph 같은 멀티에이전트 프레임워크를 즉시 도입하지 않는다.

현재 단계에서는 프레임워크보다 다음 요소를 우선한다.

- 콘텐츠 데이터 스키마 안정화
- 상태값 기반 운영 규칙 안정화
- 사람 검수 흐름 안정화
- 수동 파이프라인 1~3회 실행
- 실제 병목 확인
- 구조 검증 스크립트 유지

## Why Not Now

프레임워크는 실행 엔진이다.
하지만 현재 프로젝트는 아직 첫 콘텐츠 파이프라인을 실제로 실행해보지 않은 상태다.

지금 프레임워크를 도입하면 다음 문제가 생길 수 있다.

- 아직 검증되지 않은 운영 규칙이 코드에 고정될 수 있음
- 콘텐츠 후보, 검증, 큐레이션, 리뷰 흐름의 실제 병목을 모른 채 자동화할 수 있음
- 정책/지원금/문화행사 검증에서 사람 검수 기준이 흐려질 수 있음
- 실행 프레임워크 자체가 프로젝트 복잡도를 불필요하게 올릴 수 있음

## Current Architecture

현재 thin_wallet_club은 문서 기반 운영체계와 검증 스크립트 기반 품질관리 구조를 가진다.

- `.claude/agents/`: 역할 정의
- `.claude/skills/`: 반복 작업 절차
- `.claude/commands/`: 실행 명령 문서
- `ops/`: 운영 규칙
- `schemas/`: 데이터 계약
- `docs/`: 시스템 설명과 인수인계
- `data/`: 후보, 검증, 폐기, 성과 데이터
- `reviews/`: 사람 검수 상태 관리
- `drafts/`: 제작 확정 후 산출물
- `outputs/`: 최종 완성본
- `scripts/`: 구조/스키마/빈 폴더 검증

## Framework Candidates

### CrewAI

평가:
CrewAI는 agent, task, crew 개념이 직관적이어서 얇은지갑클럽의 역할 기반 에이전트 구조와 잘 맞는다.

장점:

- 빠른 구현 가능
- Scout, Verifier, Curator, Editor 같은 역할을 agent/task로 옮기기 쉬움
- 순차 파이프라인을 빠르게 만들기 좋음
- 사람이 중간에 개입하는 흐름을 비교적 쉽게 설계 가능

단점:

- 복잡한 상태 전환, 실패 처리, 재검증 루프에는 추가 설계가 필요
- 얇은지갑클럽처럼 검증 실패, 보류, 사람 검수, 재작업이 많은 운영에는 느슨할 수 있음

사용 시점:

- 수동 파이프라인을 1~3회 돌린 뒤 빠르게 자동화 프로토타입을 만들고 싶을 때
- 단순 순차 파이프라인을 먼저 실험하고 싶을 때

현재 결론:
1차 자동화 실험 후보.
단, 최종 실행 엔진으로 바로 확정하지 않는다.

### AutoGen

평가:
AutoGen은 대화형 멀티에이전트와 코드 실행 루프에 강점이 있다.

장점:

- 에이전트 간 대화형 문제 해결에 적합
- 코드 작성, 실행, 오류 수정 루프에 강함
- 개발 자동화 실험에 유리함

단점:

- 얇은지갑클럽의 핵심 문제는 코드 실행 루프가 아니라 콘텐츠 정보 검증, 상태 관리, 사람 검수임
- 상태 기반 운영 파이프라인에는 우선순위가 낮음
- 신규 기본 런타임으로 선택하기에는 프로젝트 목적과 다소 어긋남

사용 시점:

- 별도 코드 자동화 실험이 필요할 때
- 리서치/검증 시스템보다 개발 자동화 루프가 중요해질 때

현재 결론:
기본 선택지에서 제외.
특수한 코드 실행 실험이 필요할 때만 검토한다.

### LangGraph

평가:
LangGraph는 상태 기반 그래프, 조건 분기, 실패 처리, human-in-the-loop 설계에 강점이 있어 얇은지갑클럽의 장기 실행 엔진 후보로 가장 적합하다.

장점:

- `status_rules.md`와 잘 맞음
- candidate -> verifying -> verified -> selected -> scripted -> reviewed -> published 같은 상태 전환을 명확히 구현 가능
- 공식 출처 없음, 마감 지남, 검증 실패, 사람 검수 필요 같은 조건 분기를 설계하기 좋음
- 재검증, 보류, 폐기, 수정 요청 같은 루프를 구현하기 좋음
- 수동 파이프라인에서 얻은 실제 병목을 코드로 옮기기에 적합

단점:

- CrewAI보다 초기 구현 복잡도가 높음
- 그래프 상태, 노드, 엣지, 저장소 설계를 먼저 해야 함
- 첫 실행 엔진으로 도입하면 과설계가 될 수 있음

사용 시점:

- 수동 파이프라인을 1~3회 실행한 뒤 병목이 명확해졌을 때
- 상태 전환과 검수 루프를 자동화해야 할 때
- 장기 운영용 실행 엔진을 만들 때

현재 결론:
장기 실행 엔진 1순위 후보.
하지만 지금 당장은 구현하지 않는다.

## Adoption Criteria

아래 조건을 만족하기 전까지 멀티에이전트 프레임워크를 도입하지 않는다.

- `python scripts/validate_all.py` 통과
- `schemas/`가 실제 후보 데이터 1~3회 사용 후 안정화됨
- `status_rules.md`가 실제 운영에서 문제없이 작동함
- `human_review_rules.md` 기준으로 사람 검수를 1~3회 완료함
- 첫 콘텐츠 파이프라인을 수동으로 끝까지 실행함
- 병목이 명확히 기록됨
- 어떤 단계가 자동화되어야 하는지 명확해짐
- 자동화 대상이 단순 반복인지, 조건 분기인지 구분됨

## Future Direction

권장 순서:

1. 현재 구조 유지
2. 첫 콘텐츠 파이프라인 수동 실행
3. 1~3개 콘텐츠를 같은 방식으로 처리
4. 병목 기록
5. CrewAI로 빠른 프로토타입을 만들지, LangGraph로 상태 기반 엔진을 만들지 결정
6. 장기 실행 엔진은 LangGraph 우선 검토
7. CrewAI는 빠른 실험 브랜치에서만 검토
8. AutoGen은 기본 런타임 후보에서 제외

## Temporary Decision

현재 결정:
No Framework For Now.

잠정 우선순위:

1. Manual Pipeline
2. Structure Validation
3. Human Review
4. Bottleneck Logging
5. LangGraph Evaluation
6. CrewAI Prototype, optional
7. AutoGen, low priority

## Rule

프레임워크 도입 전에는 `engine/` 폴더를 만들지 않는다.
프레임워크 도입 전에는 CrewAI, AutoGen, LangGraph 패키지를 설치하지 않는다.
프레임워크 도입 전에는 수동 파이프라인 실행 결과를 `docs/` 또는 `reviews/`에 기록해야 한다.
